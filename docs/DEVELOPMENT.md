# Development Guide

## Prerequisites

- Node.js 20+
- pnpm (install with `npm install -g pnpm`)
- Docker and Docker Compose
- VS Code (recommended)

## Quick Setup

```bash
# Run the development setup script
./scripts/dev-setup.sh
```

Or manually:

```bash
# Start development services (PostgreSQL, MQTT, Redis)
docker compose -f docker-compose.dev.yml up -d

# Install backend dependencies
cd backend
pnpm install
pnpm prisma generate
pnpm prisma migrate dev

# Install frontend dependencies
cd ../frontend
pnpm install
```

## Running Development Servers

**Backend (Terminal 1):**
```bash
cd backend
pnpm dev
```
Backend runs at http://localhost:3000

**Scheduler (Terminal 2):**
```bash
cd backend
pnpm dev:scheduler
```
The scheduler handles:
- Marking devices as OFFLINE after 5 minutes of no heartbeat
- Cleaning up stale test runs
- UniFi auto-sync (if configured)
- VLAN validation scheduling

**Important:** The scheduler runs as a separate process. Without it, devices will remain ONLINE indefinitely even after disconnecting.

**Frontend (Terminal 3):**
```bash
cd frontend
pnpm dev
```
Frontend runs at http://localhost:5173

## Environment Variables

Copy `.env.example` to `.env` (root) and `backend/.env` (for local backend) and configure:

```env
# Database
POSTGRES_DB=zeroproof_dev
POSTGRES_PASSWORD=dev_password
DATABASE_URL=postgresql://postgres:dev_password@localhost:5432/zeroproof_dev

# MQTT
MQTT_BROKER=localhost
MQTT_PORT=1883

# Backend
NODE_ENV=development
PORT=3000
SESSION_SECRET=your-32-char-secret-here
ENCRYPTION_KEY=your-32-char-key-here
# DEFAULT_ADMIN_PASSWORD is optional. If unset, ZeroProof routes the user to
# /setup on first load to create their own admin account. Uncomment and set
# this only when seeding non-interactively (CI / scripted demos).
# DEFAULT_ADMIN_PASSWORD=
# CORS_ORIGIN=http://localhost:5173

# Frontend
VITE_API_URL=http://localhost:3000/api/v1
VITE_WS_URL=ws://localhost:3000/ws
```

The Mosquitto password file is generated by `./scripts/dev-setup.sh`. If you start Docker services manually after a fresh clone, generate it first:

```bash
docker run --rm -v "$(pwd)/mosquitto/config:/mosquitto/config" eclipse-mosquitto:2 \
  mosquitto_passwd -b -c /mosquitto/config/passwd auditor "${MQTT_PASSWORD:-mqtt_password}"
```

## UniFi Sync Notes (Docker Dev)

- Distinguish **app health** from **controller health**:
  - Backend health check: `curl http://localhost:3000/health`
  - UniFi sync can still fail while backend is healthy.
- In Docker, `127.0.0.1` / `localhost` refers to the container itself. For host services, use:
  - `HOST_IP=<your-host-lan-ip>` in `.env`, or
  - `host.docker.internal` (fallback used by backend when loopback host is configured).
- Port selection guidance:
  - UniFi OS gateways/consoles: typically `443`
  - Legacy Network Application installs: commonly `8443`
  - Tunnel ports (for example `65225`) only work while that tunnel is actively running.
- UI behavior to remember:
  - **Test Connection** uses the current form values.
  - **Sync Configuration Now** uses persisted values from `UniFiConnection`.
  - After a successful test, click **Save Settings** before syncing.
- Quick checks:
  - Host reachability: `nc -zv <controller-host> <port>`
  - Backend logs: `docker logs --since 10m zeroproof-backend-dev`

## Common Tasks

### Adding a new API endpoint

1. Create route in `backend/src/api/routes/`
2. Add validation schema in `backend/src/types/`
3. Register route in `backend/src/api/routes/index.ts`

### Adding a new analyzer

1. Create analyzer in `backend/src/analyzers/`
2. Export from `backend/src/analyzers/index.ts`
3. Add to `analyzeConfiguration()` function

### Adding a new frontend page

1. Create page in `frontend/src/pages/`
2. Add route in `frontend/src/App.tsx`
3. Add navigation item in `frontend/src/components/layout/Layout.tsx`

### Adding a new detector

Detectors are pure functions — input (windowed events) → output (`DetectionResult[]`). The runner persists results, so detectors stay easy to test.

1. Create the detector in `backend/src/detectors/<name>.ts`. Implement the `Detector` interface from `backend/src/detectors/framework.ts`.
2. Add YAML rule metadata at `rules/detection/<id>.yaml` — `id`, `severity` (with optional `escalate_to` / `when`), `remediation`, `references`, optional `retention_days`. Severity, remediation, and references can evolve without a code deploy.
3. Register the detector in `backend/src/detectors/index.ts` (both `BUILT_IN_DETECTORS` and the re-exports).
4. Write unit tests at `backend/tests/unit/detectors/<name>.test.ts`. Mock the underlying Prisma calls; do NOT hit a real database. Cover threshold tiers, dedupe, and config-driven branches.
5. Run `pnpm test --no-coverage --testPathPattern=<name>` to verify.

Detectors must be **idempotent** — running twice over the same window must produce the same fingerprints. The runner relies on the unique-fingerprint constraint to dedupe.

### Adding a new threat-intel feed

1. Create the adapter in `backend/src/services/threatIntel/feeds/<id>.ts` exporting a `createXFeed()` factory that returns an `IocFeed`. Use stdlib `node:https` — **no axios / undici / fetch deps**.
2. Register it in `backend/src/services/threatIntel/index.ts` inside `bootstrapThreatIntel()`.
3. Add a row to the "External data sources" table in `EXTERNALS.md` (URL, license, refresh cadence, why it's trustworthy).
4. Test with a sample fixture — never hit the upstream from inside a unit test.

The orchestrator handles refresh, stale-pruning, and lookups. Adapters only convert the upstream feed shape into normalized `IocRecord` objects.

### Modifying a mutating API endpoint

All mutating routes (`POST`, `PUT`, `PATCH`, `DELETE`) on `/api/v1/*` are guarded by the synchronizer-token CSRF middleware (`backend/src/api/middleware/csrf.ts`). The frontend `ApiClient` injects `X-CSRF-Token` automatically on mutating calls, so you don't need to thread it through page code. CSRF is bypassed in tests (`NODE_ENV=test`) so existing route harnesses keep working.
